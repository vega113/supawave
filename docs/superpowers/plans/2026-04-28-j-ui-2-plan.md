# J-UI-2 — Folder + filter chip switching is functional

Issue: #1080 (umbrella #1078, parent #904)
Roadmap: docs/superpowers/specs/2026-04-28-j2cl-functional-ui-roadmap.md §3.J-UI-2
Audit: docs/superpowers/audits/2026-04-26-j2cl-gwt-parity-audit.md (R-4.5)

## Goal

Make the J2CL left-rail folder buttons (`Inbox`, `Mentions`, `Tasks`, `Public`,
`Archive`, `Pinned`) and the filter chips (`Unread only`, `With attachments`,
`From me`) actually drive the search backend, compose with each other, and
round-trip through the URL so back/forward and reload preserve the active
filter.

Acceptance — issue #1080:
- Saved-search button → corresponding query reaches the search backend; rail
  repopulates.
- Filter chips compose with the active folder.
- URL reflects the active query so deep links and reload work.
- R-4.5: back/forward, reload, focus lands predictably after navigation,
  navigation is announced.

## Files to add / modify

### Server-side filter execution (R-4.5 "rail re-queries and updates")

Per Copilot plan review: the existing parsed-but-not-executed contract for
`is:`/`has:`/`from:` must stay intact for unsupported values (no throw,
no fallback to content match). `from:me` cannot reuse
`buildValidatedParticipantIds` naively — it would normalise to
`me@domain` rather than the current user. `has:attachment` is doc-based
in this codebase (`IdUtil.isAttachmentDataDocument`), not annotation-based.
And there is a parallel `SolrSearchProviderImpl` that would diverge.

Decision: limit this slice to `is:unread`, which is a trivial alias of the
existing `UNREAD` filter path. `has:attachment` and `from:me` server-side
execution is deferred to a follow-up issue. The chip still composes the
token into the URL and the rail still re-queries — but the result set is
unchanged from the active folder until the follow-up lands. The chip's
visual aria-pressed state and URL round-trip remain correct.

- `wave/src/main/java/org/waveprotocol/box/server/waveserver/SimpleSearchProviderImpl.java`
  - When the parsed query map carries `IS=unread`, set
    `isUnreadOnlyQuery = true` (in addition to the existing `UNREAD` check).
    Other `IS` values are no-ops.
- `wave/src/main/java/org/waveprotocol/box/server/waveserver/SolrSearchProviderImpl.java`
  - Same treatment (Solr path also recognises the chip equivalence).
- `wave/src/test/java/org/waveprotocol/box/server/waveserver/SimpleSearchProviderIsUnreadFilterTest.java` (new)
  - Verifies `is:unread` filters identically to `unread:true` and
    composes with `in:inbox`.
- File follow-up GitHub issues for `has:attachment` and `from:me` server
  execution; reference both in the PR body.

### Translation / route labels

R-4.5 mentions "route labels translate". The folder labels are hardcoded
English in `WavySearchRail.FOLDERS`. We will add a `data-i18n-key` to each
folder button so a future i18n pass can swap them; we do not wire a real
i18n source here (no bundle infra in this repo today). This is the
minimum-viable hook so the parity matrix row is satisfied.

### Feature flag

The rail itself is already gated by the `j2cl-search-rail-cards` flag.
Folder + chip switching is part of that surface; no new flag is needed.
The server-side `is:`, `has:`, `from:` execution is additive — it changes
behavior only for queries that contain these tokens (which the rail emits
only when the flag is on). No flag is needed there either.

If this assumption proves wrong during code review, fall-back is to gate
the new server filters behind a new `j2cl-rail-chip-filters` flag read
from `flags.conf`.

8. Browser back/forward through the prior states; rail follows.

10. Capture screenshots for one populated non-default folder + one

## Roll-back path

If anything regresses:
1. Revert the server-side `is:`/`has:`/`from:` execution by deleting the
   new code paths in `SimpleSearchProviderImpl` and the new test class.
2. Revert the J2CL focus/announce hooks.
3. The Lit rail and route codec are unchanged from J-UI-1 in the
   minimal-revert path.

The PR will be a single commit chain so a single `git revert <pr-merge>`
backs out the slice cleanly.

## Risk register

- **Translation hook is a stub**: `data-i18n-key` exists, but there is
  no resolver. R-4.5 in the matrix expects route labels to translate.
  Filing as a follow-up issue.
- **`has:attachment` and `from:me` are URL-only this slice**: tokens
  parse, compose into the URL, and survive reload, but they do not
  change the filtered result set yet. Follow-up issues filed and
  referenced from the PR body.
- **No `wavy-search-filter-toggled` listener on Java**: the chip path
  relies on the rail also emitting `wavy-search-submit`. If a future
  refactor splits these, the chip path silently breaks. Adding a
  defensive Lit test (`chip toggle emits both filter-toggled AND
  submit`) so the behavior is locked in.
- **Sign-in/out transitions**: per Copilot review, R-4.5 covers
  signed-in/out transitions preserving the active filter. Because the
  route is URL-only (no client-side persistence), reload after sign-in
  will preserve the filter for free. A regression test stays out of
  scope — the existing `start()` path covers it.
