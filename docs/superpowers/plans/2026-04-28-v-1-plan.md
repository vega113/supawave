# V-1: J2CL shell chrome rebuild — implementation plan

Issue: [#1099](https://github.com/vega113/supawave/issues/1099)
Roadmap: `docs/superpowers/specs/2026-04-28-j2cl-visual-polish-roadmap.md`
Mockup: [`docs/superpowers/mockups/2026-04-28-j2cl-functional-ui/01-shell-inbox-with-waves.svg`](../mockups/2026-04-28-j2cl-functional-ui/01-shell-inbox-with-waves.svg)
Branch: `claude/j2cl-v-1` (worktree `/private/tmp/wave-j2cl-v-1`)

## 1. Scope

V-1 is a *visual polish* slice. Acceptance is screenshot-diff vs. mockup. We
adjust SSR markup + global shell CSS so `?view=j2cl-root` composes as

- header bar (logo + brand on the left, header chrome — search / locale /
  user menu — on the right) on top
- rail (saved-search list + filter chips + digest cards) on the left
- wave panel (selected wave header + reading frame) filling the centre
- no `Plugins` rail-extension column visible on the user route
- no giant whitespace gap above the wave panel

Out of scope: V-2 (developer strings), V-3 (toolbar icons), V-4 (per-blip
chrome), V-5 (density tuning). Do not touch their files.

## 2. Concrete diffs

### 2.1 Global shell layout — `j2cl/lit/src/tokens/shell-tokens.css`

Today the global rule on `shell-root` clamps the layout to
`max-width: 1220px` with `padding: 18px 20px 24px`. The mockup is
1440x900 with the chrome flush to the viewport edges (header is full-bleed,
rail starts at x=0). Concrete edits:

- Drop `max-width: 1220px` and the outer 18/20px padding. Set the shell to
  `min-height: 100vh` filling the viewport.
- Use a 2-column grid `minmax(280px, 320px) 1fr` so the rail matches the
  296px column from the mockup and the wave panel takes the rest.
- Replace the soft outer card chrome with mockup-true surfaces:
  `shell-header` becomes a 64px full-bleed bar (no border-radius / shadow,
  hairline bottom border).
  `shell-nav-rail` becomes a left rail with 1px right hairline (no rounded
  card).
  `shell-main-region` becomes a flat surface (no rounded card / shadow).
- Tighten `gap` from 20px to 0 between header / nav / main so the chrome
  reads as one continuous shell.
- Update the brand block (`shell-header > [slot="brand"]`) to match the
  mockup: 36px circular SupaWave mark + bold "SupaWave" + small
  `J2CL · /?view=j2cl-root` eyebrow stacked.
- Eyebrow text is rendered server-side — see 2.2.

### 2.2 SSR brand markup — `wave/src/jakarta-overrides/.../HtmlRenderer.java`

Replace the dev-y brand at lines 3537 / 3643:
```html
<span aria-hidden="true">J2</span><span>SupaWave J2CL Root Shell</span>
```
with
```html
<span class="j2cl-brand-mark" aria-hidden="true">SW</span>
<span class="j2cl-brand-stack">
  <span class="j2cl-brand-name">SupaWave</span>
  <span class="j2cl-brand-eyebrow">J2CL · /?view=j2cl-root</span>
</span>
```

The "SW" mark gets the gradient pill via `.j2cl-brand-mark` styling so the
mockup's circular logo lands without an SVG asset bump (V-1 keeps assets
unchanged; the SVG mark can land in a follow-up if reviewer asks).

### 2.3 Hide the `Plugins` rail-extension column on the user route

`J2clRailExtensionParityTest` requires the `<wavy-rail-panel slot="rail-extension">`
element to be SSR'd. We satisfy the test AND the mockup by:

- Keeping the element in the DOM (so the parity test still passes — we are
  not breaking the F-4 plugin contract).
- Adding a `hidden` attribute to the empty production panel so it does not
  paint on the user route. When a plugin actually slots content into the
  rail-extension slot, the JS/Lit upgrade can remove `hidden` (existing
  flow path, no plugin currently does this in production).
- Updating the parity test only if it asserts visibility — it does not (it
  asserts presence + attributes), so no test edit needed.

Result: the rail-extension column collapses (the `1fr` track absorbs its
space because the only slotted child is `hidden`).

In `shell-tokens.css` the global grid no longer reserves a third column —
since `hidden` removes the element from layout, this is a no-op visually,
but we drop the obsolete shadow-CSS three-column grid in `shell-root.js` to
keep the source of truth in one place. Per-rule scope:

- `shell-root.js` shadow CSS: drop the third `rail` column (return to nav
  | main; status full-width row). Keep the slot in the template so the
  light-DOM element still has a render position.
- `shell-tokens.css`: keep 2-column grid as today (already only has nav |
  main).

### 2.4 Wave-panel split-layout fix — `j2cl/src/main/webapp/assets/sidecar.css`

`.sidecar-split-layout` is `grid-template-columns: minmax(320px,420px)
minmax(0,1fr)`. The first column hosts the legacy form, which has been
hidden since J-UI-5 merged; in production the column is empty and only
contributes whitespace. Change to `grid-template-columns: minmax(0, 1fr)`
when `data-j2cl-server-first-workflow="true"` is present (it always is on
the J2CL root route). Also reduce `.sidecar-selected-card` padding from
32px → 24px and drop the rounded-card chrome (border-radius 24 → 0,
shadow → none, border → 0) so the wave panel inherits the flat surface
from `shell-main-region`. Keep the inner `.sidecar-selected-content` and
blip-card chrome unchanged — V-4 owns those.

### 2.5 Top whitespace gap

Two contributors:
- `body.j2cl-root-shell-page` background gradient with rounded chrome
  creates visual top whitespace via the 18px shell padding.
- `shell-main-region` adds another 20px padding + 28px border-radius.

Both are addressed by 2.1 and 2.4. After V-1 the layout reads:
`<header 64px> | <rail | wave-panel>` with no outer gutter and no inner
card chrome.

## 5. Local-server QA — screenshot procedure

1. From this worktree: `bash scripts/worktree-boot.sh --port 9900`. If 9900
   is taken, pick the next free port from 9901-9909.
2. Open `http://localhost:<port>/auth/register`, register a fresh user
   (random email, e.g. `qa+v1+<unix-ts>@wavelylabs.com`).
3. Sign in, navigate to `?view=j2cl-root`.
4. Capture `after.png` at 1440×900 viewport.
5. Switch to a sibling main-pinned worktree (or `git stash` the v-1 changes
   and `git checkout origin/main` in a separate clone) and capture
   `before.png` with the same user / same viewport.
6. Save both under `docs/superpowers/screenshots/v-1/`.
7. Compare to `01-shell-inbox-with-waves.svg`. If after-screenshot is
   <85% mockup-fidelity, iterate.
8. Edge cases: empty inbox, narrow viewport (768px), signed-out
   (`?view=j2cl-root` while signed out), `?view=gwt` (must be unaffected).

If `claude-in-chrome` MCP is available, drive screenshots via it.
Otherwise spawn a small playwright node script under `scripts/qa/v1-screenshot.mjs`.

## 6.1 Plan revisions from copilot review

- `wavy-rail-panel.js` does not currently have a `:host([hidden]) { display: none }`
  rule (it sets `:host { display: block }`, which the `hidden` attribute
  would override but only because Lit-rendered hosts inherit user-agent
  `[hidden]` semantics; verified by the explicit `:host([hidden])` rules
  used by `wavy-floating-scroll-to-new`, `wavy-version-history`, and
  `wavy-profile-overlay`). Add the rule explicitly to keep the contract
  source-visible.
- Header chrome already includes a right-side `<wavy-header>` cluster
  (locale picker, bell, mail, user-menu trigger) plus a duplicate
  `.brand` link that mirrors the outer `slot="brand"`. Hide the
  duplicate with a global selector rather than mutating the SSR (would
  break parity tests).
- The mockup's centered global search input is owned by the rail's
  query box today (B.1 — wired by F-2 slice 5). Relocating the query
  box to the header is a wiring change, not a CSS change; document
  this gap in the PR body and scope it as a follow-up. V-1 still
  matches the rail+wave-panel composition + header chrome the issue
  enumerates.
- Mockup's "+ New Wave" CTA + avatar pill on the right are already
  emitted by `wavy-header` (avatar) and `wavy-search-rail` (B.3 New
  Wave). The new-wave button stays in the rail for V-1 (relocating it
  to the header is the same wiring class as moving search; mark as
  follow-up).
- Brand mark: ship the mockup's `<symbol id="supawave-mark">` inline as
  a 36px SVG inside `slot="brand"` instead of a "SW" gradient pill.
- Changelog fragment lives at `wave/config/changelog.d/<date>-<slug>.json`
  with `{releaseId, version:"PR #NNNN", date, title, summary, sections:[{type, items}]}`
  schema (validated by `scripts/validate-changelog.py`). Use
  `wave/config/changelog.d/2026-04-28-v1-j2cl-shell-chrome.json`,
  patch `version` after PR creation.
- Mobile (<860px): keep the existing single-column collapse from
  `shell-tokens.css`; the reduced outer padding still works at 360px.
- `?view=gwt` regression: that route renders via `WaveClientServlet`,
  not the J2CL root shell — none of the V-1 files affect it. Verify
  via a quick screenshot in QA.
