# V-3 Plan — Real format toolbar with icons

## 1. Goal

Replace the 22-button text-labelled format toolbar at
`?view=j2cl-root` with a compact icon row matching
`03-inline-rich-text-composer.svg` (lines 121-189). Each button renders as
an SVG icon (no text content) with `title` / `aria-label` for
accessibility. Selection-driven pressed state lights with the cyan accent
when the corresponding annotation is in the active selection.

This is **visual polish only**. Action emission, the
`wavy-format-toolbar-action` event contract, the
`DAILY_RICH_EDIT_ACTION_IDS` export, and the existing test surface in
`j2cl/lit/test/wavy-format-toolbar.test.js` and
`j2cl/lit/test/toolbar-button.test.js` must continue to pass.

## 3. Mockup decode

From `03-inline-rich-text-composer.svg` lines 121-189:

- Toolbar is a single dark pill (`#0b1320` fill, 540×48, radius 12,
  shadow under). Anchored above the selected text fragment.
- Icon tiles are 32×32 with 4px padding inside the pill, radius 8.
- Active tile: cyan (`#22d3ee`) fill, dark glyph.
- Inactive tile: transparent fill, white glyph.
- Group separators: 1px white@20% vertical lines.
- Groups (left → right):
  1. **typography**: `B`, `I`, `U`, `S`
  2. **block**: `H1`, `H2`, `"` (blockquote)
  3. **lists**: bulleted, numbered
  4. **alignment**: align-left, align-center (right + RTL implied by issue)
  5. **link / code**: link, `{ }` (code)

The issue body refines the group set we ship:

- Text: bold / italic / underline / strikethrough
- Block: heading / bulleted / numbered / blockquote
- Alignment: left / center / right / RTL
- Link: create / remove
- Clear formatting

I will treat "code" and the second heading level (H2) in the mockup as
out of scope (no actionId exists, and the issue group list doesn't
include them). H1/H2 in the mockup collapse to the single existing
`heading` action.

## 4. File-level plan

### 4.A. New: `j2cl/lit/src/icons/toolbar-icons.js`

Single module exporting one Lit `svg`-tagged template per action id.
Each icon is sized for a 16×16 viewBox so the 32×32 tile keeps 8px
padding around the glyph. White current-color stroke / fill so the
parent button can flip color via CSS variables on
`aria-pressed=true`.

Icons to author (one per action id):

- `bold`, `italic`, `underline`, `strikethrough`
- `heading` (H with bar), `unordered-list`, `ordered-list`,
  `blockquote`-equivalent (we will not ship blockquote glyph — see
  §5 — `indent`/`outdent` keep their existing actionIds and get small
  bracket glyphs)
- `align-left`, `align-center`, `align-right`, `rtl`
- `link`, `unlink`
- `clear-formatting`
- `insert-task` (small checkbox)
- `attachment-insert` (paperclip)

Icons sourced from a hand-rolled set inside the repo — no new npm
dependency. Each glyph is 6-10 SVG path/line elements; total module
size budget < 6 KB raw.

### 4.B. Edit: `j2cl/lit/src/elements/toolbar-button.js`

Extend with an `icon` property (string, action id). When set:

- Render the matching icon template inside the `<button>` instead of
  the text label.
- Keep `label` populated and bind it to both `aria-label` **and `title`**
  attributes — neither is present on the current button render, so this
  is a real diff (not a no-op as the prior plan revision implied).
- Default-export render path stays text-only when `icon` is unset, so
  every existing consumer (e.g. `wave-blip-toolbar.js`) continues to
  render text. No regression to non-V-3 surfaces.

CSS:

- 32×32 tile via `width`/`height` on the `<button>` when
  `icon` is set; padding collapses; border-radius 8.
- `aria-pressed=true` background flips to
  `var(--wavy-toolbar-tile-active-bg)`, glyph color flips to
  `var(--wavy-toolbar-tile-active-fg)`.
- `:hover:not(:disabled)` background lifts to
  `rgba(255,255,255,0.08)` so non-toggle buttons (heading / align /
  link / clear) still give click feedback. The mockup omits hover
  styling but the live UX needs it.
- `:focus-visible` carries `var(--wavy-focus-ring)` so keyboard users
  see selection.

### 4.C. Edit: `j2cl/lit/src/elements/wavy-format-toolbar.js`

- **Structural change** (call out): the current `render()` flat-maps
  every button into the toolbar. V-3 wraps each group into its own
  `<toolbar-group>` with an explicit `<span class="toolbar-divider"
  aria-hidden="true"></span>` sibling between groups. Existing tests
  use `querySelectorAll('[data-toolbar-action]')` which crosses
  shadow-light boundaries the same way regardless of nesting depth, so
  they continue to pass — but the DOM shape changes, so a snapshot
  test would catch the change. We add one.
- Pass `icon=<actionId>` and `label=<existing label>` to each
  `<toolbar-button>`.
- Keep `data-toolbar-action`, `action`, `pressed`, `toggle` attributes
  exactly as today so the existing tests in
  `j2cl/lit/test/wavy-format-toolbar.test.js` continue to find buttons
  by `[data-toolbar-action="…"]` and assert the pressed semantics.
- Re-order `DAILY_RICH_EDIT_ACTIONS` so the visual left-to-right groups
  match the mockup. `insert-task` and `attachment-insert` keep their
  existing actionIds (per the issue's acceptance set + the
  attachment-insert test) and live in a trailing `insert` group after
  the link group; the mockup excerpt does not show them but the
  surrounding sprint already requires their presence (V-1
  reorganisation may move them later — out of scope here).
- Explicit dividers (`<span class="toolbar-divider">`) emitted between
  groups by `wavy-format-toolbar.js` — **not** via a `::slotted`
  sibling selector on the inner toolbar div, which doesn't reliably
  match light-DOM siblings inside the slot.

### 4.D. Edit: `j2cl/lit/src/design/wavy-tokens.css`

- Add toolbar layout tokens (theme-invariant unless otherwise noted):
  - `--wavy-toolbar-pill-bg: #0b1320` — pin the dark pill across all
    themes per the Stitch/mockup direction. `--wavy-bg-surface` flips
    in light/contrast modes so we cannot reuse it. Defined at `:root`
    only; light/contrast variants do not override.
  - `--wavy-toolbar-pill-fg: rgba(255, 255, 255, 0.92)` — same
    rationale.
  - `--wavy-toolbar-tile-size: 32px`
  - `--wavy-toolbar-tile-radius: 8px`
  - `--wavy-toolbar-divider: rgba(255, 255, 255, 0.20)`
  - `--wavy-toolbar-tile-active-bg: var(--wavy-signal-cyan)`
  - `--wavy-toolbar-tile-active-fg: #0b1320`
  - `--wavy-toolbar-tile-hover-bg: rgba(255, 255, 255, 0.08)`
- These are referenced from `wavy-edit-toolbar.js` and `toolbar-button.js`.
  Theme overrides for `wavy-bg-surface` / `wavy-text-body` no longer
  flip the pill, which keeps the toolbar look consistent across light /
  dark / contrast modes — explicitly traded against the existing
  theme-aware behavior since the mockup pin is unambiguous.

### 4.E. Edit: `j2cl/lit/src/design/wavy-edit-toolbar.js`

- Tighten gap to 0 inside the dark pill (groups own their own internal
  gap; explicit `toolbar-divider` siblings supply the visible
  separators).
- Reduce padding to `var(--wavy-spacing-1)` so the 32px tile sits flush
  against the pill edge as in the mockup.
- Switch background/foreground from `--wavy-bg-surface` /
  `--wavy-text-body` to `--wavy-toolbar-pill-bg` /
  `--wavy-toolbar-pill-fg` so the pill stays dark in light/contrast
  themes (per copilot review feedback + Stitch mockup pin).
- Style the slotted toolbar-divider:
  `::slotted(.toolbar-divider) { width: 1px; height: 24px;
   background: var(--wavy-toolbar-divider); margin: 0
   var(--wavy-spacing-2); }`. Using `::slotted` on a single class is
  reliable; the brittle case the copilot review flagged was
  `::slotted(toolbar-group + toolbar-group)` which depends on sibling
  order across the slot boundary. Class-based selection sidesteps
  that.

### 4.F. Test updates: `j2cl/lit/test/`

- `toolbar-button.test.js`: add a case asserting that when `icon=bold`
  is set, the button renders an `<svg>` and the text label is hidden
  while `aria-label` and `title` still match `label`.
- `wavy-format-toolbar.test.js`: keep all existing assertions (action
  ids, attachment-insert, insert-task, descriptor → pressed). Add:
  - one case asserting every toolbar-button rendered by
    `<wavy-format-toolbar>` contains an `<svg>` element (per copilot
    review — guards silent regression when the icon module changes).
  - one case asserting at least one `.toolbar-divider` renders
    between groups (so dropping the explicit divider mistake fails
    visibly).

## 8. Edge cases to verify in QA

- Empty composer body → toolbar stays hidden (selection collapsed).
- Narrow viewport (~600px) → toolbar wraps gracefully or scrolls
  horizontally; no overflow clipping the right-most icon.
- Light theme — verify the active-tile cyan fill remains legible
  against the light pill background. (The pill is dark in the mockup
  and stays dark in our implementation regardless of theme to match the
  Stitch design.)
- High-contrast mode — focus ring on tab-focused tile is visible.
- GWT route (`?view=gwt`) is not affected: the legacy GWT toolbar
  isn't built from these Lit elements at all, so a Lit-only icon swap
  cannot regress it. Verify by visiting `?view=gwt` and confirming the
  classic toolbar renders unchanged.

## 11. Risk + rollback

- Risk: existing consumers of `<toolbar-button>` (e.g.
  `wave-blip-toolbar.js`) might pick up unintended icon rendering.
  Mitigation: `icon` is an opt-in property; default-empty
  consumers render unchanged.
- Risk: SVG-in-shadow-DOM color inheritance. Mitigation: use
  `currentColor` everywhere in icon paths so the parent
  `<button>`'s `color` propagates.
- Rollback: revert this commit; toolbar text labels return immediately.
