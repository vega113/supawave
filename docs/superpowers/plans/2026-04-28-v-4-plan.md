## 1. Scope

Visual polish only — functional wiring (focus frame, j/k nav, thread
collapse, task toggle, reply/edit/delete/link) is already in place from
J-UI-1..J-UI-8. This slice closes the screenshot gap between the live
J2CL surface and the `02-open-wave-threaded-reading.svg` mockup.

## 2. What the mockup specifies (per blip)

1. **Author avatar** — colored disc with 2-letter initials. Per-author
   color (cyan / violet / amber / slate in mockup). Root blip: 40px
   diameter; reply: 32px.
2. **Author display name** — bold, h3 weight, beside avatar.
3. **Relative timestamp** — quiet meta text after author. Root blip
   includes `· root` suffix.
4. **Per-blip toolbar (on hover/focus)** — pill-shaped white background
   with 1px hairline, glyph-prefixed entries arranged in a row:
   `↩ Reply | ✎ Edit | 🗑 Delete | 🔗 Link | ✓ Task`, with vertical
   dividers between each entry.
5. **Focus frame** — focused blip wraps in a card with cyan 2px border
   + cyan glow shadow. Toolbar inside the focused card paints with a
   cyan-soft fill instead of white.
6. **Thread chevron** — `▾` (open) / `▸` (collapsed) glyph at the left
   gutter of each blip with replies. (Functional collapse already
   wired in J-UI-4 — we just need the glyph rendered.)
7. **Indent guides** — vertical cyan-soft line down the gutter of each
   thread depth.
8. **NEW pill** — small cyan badge after the timestamp on the most
   recent blip (already exists for unread but as a dot — mockup shows
   text pill).

## 5. Test strategy

- **Lit unit (Vitest)**: extend `wavy-blip-card.test.js` to assert
  presence of new tokens, deterministic avatar palette index, chevron
  presence when `replyCount > 0`, no chevron at `replyCount = 0`.
- **Java unit**: `J2clReadSurfaceDomRendererTest` already covers
  attribute emission. No new java tests; depth attribute is a 1-line
  addition.
- **sbt compile** + **sbt test j2cl/test**.

## 6. Screenshot procedure

1. Boot worktree:
   `bash scripts/worktree-boot.sh --port 9904` (free port).
2. Register a fresh user (do not assume `vega`).
3. Sign in, create a wave, post 2-3 replies as the same user (single
   author is fine for V-4 — palette is per-id, can be observed by
   re-signing-in as a second user; primary acceptance is the chrome,
   not multi-author colors).
4. Navigate to `?view=j2cl-root`, open the wave, focus a blip with
   `j`/`k`.
5. Capture **after** screenshot at 1440x900 → `docs/superpowers/screenshots/v-4/after.png`.
6. From a sibling worktree pinned to `main`, repeat steps 1-4 and
   capture **before** screenshot → `docs/superpowers/screenshots/v-4/before.png`.
7. If `claude-in-chrome` MCP is rate-limited, fall back to a
   `puppeteer` / `playwright` node helper under `scripts/` that
   logs in via a dev-token cookie. If that is unavailable too,
   ask user to capture.
8. Reject and iterate if fidelity vs. mockup < 85%.
9. Edge cases: empty wave (no blips), narrow viewport (768px),
   signed-out (`?view=j2cl-root` should redirect / show signed-out
   shell), GWT path (`?view=gwt`) unaffected.

## 7. Feature flag

Visual polish that affects only `?view=j2cl-root` does not need a
separate flag — the J2CL route is itself behind the `j2cl-read-surface`
flag, and changing visuals on that route is the explicit slice goal.
We will not introduce `j2cl-blip-chrome` as a new flag; the slice
acceptance is "the J2CL surface matches the mockup."

## 9. Risks / non-goals

- **Out of scope**: thread-collapse functionality (already J-UI-4),
  reactions visual polish, dark-mode polish (V-1 owns shell layout
  light/dark sweep), mobile (deferred), animation motion changes.
- **Risk**: avatar palette must remain stable across re-renders so
  the same author keeps the same color in a session. Hash uses
  `author-id`, not display name (display names can change).
- **Risk**: the mockup uses emoji glyphs (🗑, 🔗); fall back to plain
  Unicode where rendering is platform-dependent. Use SVG where the
  emoji renders inconsistently — defer to a follow-up if needed.

## 10. Done criteria

- All unit tests pass.
