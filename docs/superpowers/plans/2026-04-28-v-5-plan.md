## 1. Goal

Final pass over proportions and whitespace per `01-shell-inbox-with-waves.svg`:

- Tighten the wave-list digest card (`wavy-search-rail-card`) so visible cards stack closer to the mockup's 92–120 px range instead of the current ~150 px outcome with 12 px padding + 3-line snippet bleed.
- Widen the centre wave panel by trimming the rail's max width and the wave panel's inner gutter.
- Remove the visible top whitespace gap on `?view=j2cl-root` between the 64 px header bar and the first content row of the wave panel.
- Wire the rail / pane width values through `wavy-tokens.css` so future slices can tune them centrally.

## 2. Mockup measurements (1440 × 900)

| Region              | Mockup px | Mockup % |
|---------------------|-----------|----------|
| Header              | 1440 × 64 | 100 / 7  |
| Nav rail            | 296       | 20.5 %   |
| Search-result panel | 448       | 31.1 %   |
| Wave panel          | 696       | 48.3 %   |

Digest cards in the mockup: 120 px (selected w/ 3-line snippet), 100 px (2-line), 92 px (1-line). Card-to-card vertical gap: 16 px. Internal padding: 16 px.

The current Lit shell collapses the rail and the search-result panel into one column (the rail SSR-renders the digest cards inline). This slice does not re-introduce the third column — that is V-1's territory. V-5 only tunes the *density* inside the existing two-column shell so the proportions and breathing room land within ±10 % of the mockup.

## 3. Concrete changes

| Change | File | Reason |
|---|---|---|
| Add `--wavy-rail-card-padding-y/x`, `--wavy-rail-card-gap`, `--wavy-rail-card-snippet-mb`, `--wavy-rail-width`, `--wavy-pane-gutter` density tokens | `j2cl/lit/src/design/wavy-tokens.css` | Single source of truth for V-5's density values; future slices can override per breakpoint. |
| Switch `:host` padding / margin / inner gap on the digest card to consume the new tokens; tighten visual gap to 12 px and snippet→footer gap to 6 px | `j2cl/lit/src/elements/wavy-search-rail-card.js` | Reduces the rendered card height when the snippet wraps to 1–2 lines (most cards), bringing visible cards into the 92–120 px target. |
| Narrow the rail grid column to `minmax(248px, 280px)` and trim the wave panel inner inset on the server-first workflow from 24 → 12 px top / 16 px sides | `j2cl/lit/src/tokens/shell-tokens.css`, `j2cl/src/main/webapp/assets/sidecar.css` | Widens the wave panel by ~32 px on a 1440 viewport and removes the perceptible top-of-pane gap between the 64 px header bar and "OPENED WAVE". |
| (Same `shell-tokens.css` rule) seed the rail width from `--wavy-rail-width` so density can flow from `wavy-tokens.css` if a future slice needs it | `j2cl/lit/src/tokens/shell-tokens.css` | Density tokens are only useful if both layers consume them. |

No JS behavior changes; this slice is CSS-only plus a token addition.

## 4. Acceptance contract

1. **Mockup ref** — `01-shell-inbox-with-waves.svg`.
2. **Before screenshot** — captured against `origin/main` HEAD on `?view=j2cl-root`, signed in as a freshly registered QA user, 1440 × 900 viewport.
3. **After screenshot** — captured on this branch with the same registration and viewport.
4. **Side-by-side annotation** — measurements: rail width / wave panel width / digest card height / top-of-pane gap, before/after, vs. the mockup.

A reviewer is allowed to reject the PR purely on visual fidelity even if every test passes.

## 5. Test strategy

- `cd j2cl/lit && npm test` — exercise every Lit unit, including `wavy-search-rail-card.test.js` (the existing 3-line clamp contract is unchanged so the `-webkit-line-clamp: 3` assertion stays green).
- `sbt 'wave/Test/compile'` + `sbt 'wave/testOnly *J2cl*Parity* *HtmlRendererJ2clRootShell*'` — confirm none of the parity / HTML-renderer assertions break on the token / CSS edits.
- Local server boot via `scripts/worktree-boot.sh --port <free>`, register a fresh `qa…@local.net` user, navigate to `?view=j2cl-root`, capture the after screenshot. Capture the before screenshot from a sibling worktree pinned to `origin/main` (or by stashing the V-5 changes and re-running the same boot path).
- Confirm `?view=gwt` still returns 200 and is unaffected (V-5 only touches J2CL Lit + the J2CL-only sidecar CSS).

## 6. Out of scope

- Animations / motion polish (umbrella deferral).
- Mobile / dark-mode polish (separate mockups; not in this sprint).
- Splitting the rail back into a third column (V-1's territory).
- Per-blip avatar / display-name chrome (V-4).
- Toolbar icons (V-3).

## 7. Feature flag

Not flagged. The density change is a CSS tightening on the existing `?view=j2cl-root` route which is itself the J2CL-shell flag's territory; gating density behind a second flag would force every reviewer to flip two switches to see the same product surface. The change degrades cleanly: if any token is unset the `var(--…, fallback)` declarations fall back to the prior values.
