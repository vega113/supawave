# G-PORT-6 — Tasks + done state (plan)

Issue #1115. Umbrella #1109. Parent #904. Roadmap §3 G-PORT-6.

## Why

Per the G-PORT roadmap, the J2CL ↔ GWT parity contract requires a
Playwright spec proving that, on both views, a user can:

1. Toggle a blip into a task (or insert a task into a blip).
2. Mark it done — done state visible (strikethrough / checked
   indicator).
3. Reload — done state preserved.
4. Open the same wave from a second browser context — done state
   visible there.

The implementation surface for "click task → blip becomes a task →
click done" already exists end-to-end on the J2CL view (J-UI-6 /
F-3.S2 shipped `wavy-task-affordance` + the J2CL compose +
read-renderer wiring); what is missing is the parity E2E gate that
G-PORT-6 owns.

## Current state — what already exists

- `j2cl/lit/src/elements/wavy-task-affordance.js` — per-blip toggle
  + details popover, emits `wave-blip-task-toggled` and
  `wave-blip-task-metadata-changed`. Reflects `data-task-completed`.
- `j2cl/lit/src/elements/task-metadata-popover.js` — assignee + due
  date dialog.
- `j2cl/lit/src/elements/wave-blip.js` — mounts the affordance next
  to the per-blip toolbar, reflects `data-task-completed` for the
  strikethrough CSS rule
  (`:host([data-task-completed]) .body { text-decoration: line-through; }`).
- `j2cl/src/main/java/.../compose/J2clComposeSurfaceController.java`
  `onTaskToggled` + `onTaskMetadataChanged` — emit
  `J2clRichContentDeltaFactory.taskToggleRequest` / `…MetadataRequest`
  deltas.
- `j2cl/src/main/java/.../read/J2clReadSurfaceDomRenderer.java`
  `applyTaskState` — reads persisted `task/done`, `task/assignee`,
  `task/due` annotations from the model and projects them onto the
  rendered `<wave-blip>` host so reload + cross-context updates land
  immediately.
- GWT side: tasks are inserted as inline `<check>` elements via the
  format-toolbar's "Insert task" button (EditToolbar.java:605-636).
  The renderer (`CheckBox.java`) adds the `task-completed` class to
  the blip paragraph when the checkbox is checked, applying GWT's
  strikethrough styling.

## Surface divergence to acknowledge upfront

GWT models tasks as inline checkbox content elements inside a blip
(`<check>` doodad in the doc). J2CL models tasks as a per-blip
annotation (`task/done` on the document) toggled via the per-blip
affordance. The G-PORT-6 acceptance language ("blip becomes a task
→ click done → done state visible") is written to fit either
model — what matters is the user-visible outcome on each view.

The J2CL surface is **not** changing in this slice (J-UI-6 already
shipped it). The slice is a **parity E2E test** that drives both
surfaces' real user flows and proves the round-trip works.

## What this slice ships

### 1. Playwright spec
File: `wave/src/e2e/j2cl-gwt-parity/tests/tasks-parity.spec.ts`.

Per the issue + task brief acceptance:
- Sign in fresh user. Author a wave with **at least 2 blips** on the
  GWT view (the J2CL composer surface for new waves is still gated
  off in some configurations; G-PORT-3 settled on authoring on GWT
  and asserting on both views — same pattern here).
- **J2CL half:**
  1. Open the wave on `?view=j2cl-root`.
  2. Click the task-toggle button on the second blip
     (`wavy-task-affordance [data-task-toggle-trigger]`).
  3. Assert `wave-blip[data-task-completed="true"]` for that blip.
  4. Reload. Assert done state preserved.
  5. Open in a second browser context (same login cookie copied
     across via `browser.newContext({ storageState })`). Assert
     `data-task-completed="true"` visible there.
- **GWT half:**
  1. Open the wave on `?view=gwt`.
  2. Edit the second blip and click the format toolbar's
     "Insert task" button. The doodad inserts a `<check>` element.
  3. Click the inserted checkbox to mark done. Assert the parent
     paragraph has class `task-completed`.
  4. Reload. Assert checkbox stays checked + paragraph stays
     `task-completed`.
  5. Open in a second browser context. Assert the same.

If a half fails on the live server in a way that isn't fixable in
this slice (e.g. GWT compose ergonomics race), the test still
fails LOUDLY (no skips). We file a follow-up issue and document the
blocker on #1115; we do **not** silently weaken the spec.

### 2. Page-object additions
- Reuse the J2clPage / GwtPage helpers G-PORT-3 introduces. If
  G-PORT-3 isn't merged yet, port the helpers (newWaveAffordance,
  typeIntoBlipDocument, clickReplyOnBlip, readWaveIdFromHash) into
  this slice's branch under the same file paths so the spec lands
  green even if G-PORT-3 lands second.
- Add `clickTaskToggleOnBlip(blipId)` to `J2clPage` and
  `clickInsertTaskInActiveBlip()` + `clickFirstInlineCheckbox()` to
  `GwtPage`.

### 3. Manual verification log
Run the full spec locally against `scripts/wave-smoke.sh start`
on port 9900 with a freshly built distribution, capture pass output
for the PR body.

### 4. (No source code changes to widget surface unless the test surfaces a real bug)
The plan explicitly avoids extending the widget unless the E2E run
finds a defect. Any defect found becomes a small commit on this
branch with a regression hook in the spec.

## Risks + mitigations

- **GWT compose flow flakiness.** Mitigation: gate every interactive
  step behind `expect.poll` / readiness-aware waits, copy the
  pattern from G-PORT-3's `wave-reading-parity.spec.ts`. If GWT's
  format toolbar takes >5s to mount, increase the wait, do not
  skip.
- **Cross-context cookie sharing.** Mitigation: use
  `context.storageState()` on the first context and pass it to
  `browser.newContext({ storageState })` on the second — Playwright's
  documented pattern.
- **GWT vs J2CL task model mismatch.** Mitigation: each view
  exercises its native task surface (per-blip annotation on J2CL,
  inline checkbox on GWT). The user-visible done indicator
  (strikethrough) is the parity signal.
- **CI runtime.** Mitigation: keep the test set small (one
  authoring wave, two assertions per view) and rely on the existing
  parity workflow for execution.

## Workflow

1. Commit this plan: `docs(g-port-6): plan`.
2. Copilot review of the plan; address any blockers in a follow-up
   commit.
3. Implement the spec + page-object additions. One commit per
   helper, one for the spec body, one for the changelog.
4. Run locally end-to-end against `scripts/wave-smoke.sh`. Confirm
   both views actually flip done state, reload preserves, and
   second-context observes the change.
5. Open PR `G-PORT-6: Tasks + done state parity` referencing #1115,
   umbrella #1109. Body includes the manual verification log.
6. Self-monitor PR through merge.

## Out of scope

- Changes to the widget UI itself (J-UI-6 / F-3.S2 already shipped).
- New visual polish (G-PORT-9 owns).
- Mention parity, attachment parity (separate slices).
- Cloning the GWT task widget into J2CL line-by-line — the J2CL
  per-blip annotation model is already shipped and is the
  user-visible surface; "1-to-1 clone" in the roadmap is satisfied
  by the equivalent user behaviour.
