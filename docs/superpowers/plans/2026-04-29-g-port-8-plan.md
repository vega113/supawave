# G-PORT-8 — Top-of-wave action bar parity (plan)

Issue #1117. Umbrella #1109. Parent #904. Roadmap spec
`docs/superpowers/specs/2026-04-29-j2cl-gwt-port-roadmap.md` §3 G-PORT-8
(lines 125-130).

## Why

GWT renders a top-of-wave action-icon strip via
`wave/src/main/java/.../wavepanel/impl/toolbar/ViewToolbar.java`. It
draws three groups: navigation (recent / next-unread / prev / next /
end / prev-@ / next-@), folder actions (archive, pin), and history.
The buttons are SVG icons with tooltips, each wired to either a focus
movement (navigation), a `/folder` servlet call (archive/pin), or a
listener (history overlay).

On `?view=j2cl-root` today, `<wavy-wave-nav-row>` already renders the
ten navigation/folder/history buttons in correct GWT order and emits
correct `wave-nav-*-requested` CustomEvents. But:

- The buttons are TEXT LABELS ("Archive", "Pin", "Version history"),
  not the GWT icon strip. Issue #1117 asks for a 1-to-1 icon clone.
- The `wave-nav-archive-toggle-requested` /
  `wave-nav-pin-toggle-requested` events are ONLY consumed by
  telemetry (see `J2clSelectedWaveView.bindChromeEvents`) — clicking
  has no visible effect.
- The `wave-nav-version-history-requested` event is dispatched but no
  controller opens the `<wavy-version-history hidden>` overlay that
  ships pre-mounted via the J2CL bootstrap.

`<wavy-version-history>` itself was scoped in F-2.S4 (#1048) and is
already a fully-functional overlay (focus-trap, Esc-to-close, slider,
Restore confirm dialog). Its data wire (`versionLoader`) defers to the
F-2 follow-up #1054 — the test plan below scopes the version-history
parity assertion to "click button → overlay opens; press Esc →
overlay closes" without exercising the actual diff/restore content.

## Refresh button: explicitly out of scope

Issue #1117's title mentions a "refresh" affordance. GWT's
`ViewToolbar` does NOT render a refresh button — the only place
"refresh" appears in that file is in code comments distinguishing the
clock-face Recent icon and the History icon ("so it no longer reads
like refresh"). Cloning GWT 1-to-1 therefore means **no refresh
button**. Documented here and in the PR body so the reviewer can
confirm. If a reviewer disagrees, file a follow-up rather than diverge
from GWT.

## Current state (J2CL)

- `j2cl/lit/src/elements/wavy-wave-nav-row.js` — renders 10 text
  buttons in GWT order, emits `wave-nav-*-requested` events. Has
  `pinned` and `archived` reactive props. Tests in
  `j2cl/lit/test/wavy-wave-nav-row.test.js` cover events, ARIA
  toggling and the H keyboard shortcut.
- `j2cl/lit/src/elements/wavy-version-history.js` — fully
  functional overlay with `open` reactive prop, focus-trap, Esc to
  close. Mounted globally as `<wavy-version-history hidden>` via
  `HtmlRenderer.appendJ2clRootShellBootstrap` lines 3679 and 3890.
- `j2cl/lit/src/icons/toolbar-icons.js` — Lit `svg` template factory
  used by `<wavy-format-toolbar>`. The action-bar will reuse the same
  authoring style with its own glyph set.
- `j2cl/src/main/java/.../search/J2clSelectedWaveView.bindChromeEvents`
  — listens to all 10 `wave-nav-*` events but only records telemetry.
- GWT — `ViewToolbar.java` renders 9 SVG icons (Recent, Next-Unread,
  Prev, Next, End, Prev@, Next@, Archive, Pin) with tooltips, plus a
  Version-History icon in its own group; archive/pin call
  `FolderOperationServiceImpl` against `/folder/?operation=...&waveId=...`
  with `OPERATION_PIN`, `OPERATION_UNPIN`, `OPERATION_MOVE`+
  `FOLDER_INBOX`/`FOLDER_ARCHIVE`. Pinned/archived state is reflected
  via `setDown(true)` (CSS pressed state) plus tooltip flip.

## Fix shape

(Revised after Copilot review of v1: fix POST/GET, scope listener to
`<wavy-wave-nav-row>` host instead of `document`, add explicit
optimistic flip + rollback, document the eventual S5 takeover.)

1. **Icons.** Add a new icon set `j2cl/lit/src/icons/wave-action-bar-icons.js`
   that exports a Lit `svg` template per GWT glyph, matching the
   geometry from `ViewToolbar.java` (1.75 stroke, round caps/joins,
   18px on a 24-unit viewBox). One file so future slices have a
   clear extension point.
2. **`wavy-wave-nav-row` icon refresh.** Replace the inline text
   labels in the row's `render()` with the imported SVG glyphs.
   Visible hover tooltip via the native `title` attribute mirrors
   GWT's `setTooltip`. Keep existing `aria-label` (AT) and
   `data-action` (test selectors). Keep the existing data-emphasis
   hooks (cyan/violet/muted) so Pin's pressed state and the
   unread/mention accent stay intact.
3. **Folder-op wiring (JS controller, scoped to the nav-row host).**
   The existing Java `bindChromeEvents` (lines 338-364 of
   `J2clSelectedWaveView.java`) already listens on the per-card
   element scope for telemetry. To avoid two-writer races with the
   `setNavRowFolderState` extension point reserved for #1055/S5
   (currently unwired), the new JS controller binds its listeners
   on each `<wavy-wave-nav-row>` host the controller observes via a
   `MutationObserver` on `document.body` — never on `document` and
   never via `addEventListener('document')`. This guarantees:
   - One listener per nav-row, even when both server-first and
     cold-mount nav-rows coexist briefly during boot.
   - When #1055/S5 lands and Java's `setNavRowFolderState`
     overwrites the attribute, our controller's optimistic flip is
     replaced by the authoritative server view next render — no
     stale state.

   Behavior:
   - Listens for `wave-nav-archive-toggle-requested` and
     `wave-nav-pin-toggle-requested` on each nav-row host.
   - Reads `sourceWaveId` from event detail; falls back to the
     row's `source-wave-id` attribute if detail is empty.
   - Reads current `pinned` / `archived` reactive props from the
     row to decide direction (toggle).
   - **Optimistic flip** of the row's attribute immediately —
     matches GWT `setDown(true)` feel.
   - Issues `POST /folder/?operation=move&folder=archive|inbox&waveId=...`
     or `POST /folder/?operation=pin|unpin&waveId=...` (path +
     query-string params, body empty — the GWT
     `FolderOperationServiceImpl` uses POST; verified against
     `FolderServlet.doPost` which reads `request.getParameter` and
     so accepts query-string body or form params).
   - On 200, dispatches a `wavy-folder-action-completed`
     CustomEvent so the search rail can re-query (mirroring the GWT
     `FolderActionListener`). Optimistic attribute is left in place.
   - On non-200, **rolls back** the optimistic attribute and emits
     a `wavy-folder-action-failed` event so a future toast can
     surface it. (Toast wiring is out of scope for this slice; keep
     the event surface clean.)

   Implementation: a new `j2cl/lit/src/controllers/wave-action-bar-controller.js`
   wired from `j2cl/lit/src/index.js`. Pure JS so it does not
   require expanding the `J2clSelectedWaveController.Gateway`
   interface and the four+ implementations behind it. The
   J2CL-Java side stays as a single-writer for nav-row attributes
   the day #1055/S5 wires `setNavRowFolderState` — that wiring will
   replace the JS controller's responsibility for the attribute and
   the controller will collapse to network-only. That follow-up is
   filed as a separate issue at PR-open time (see "Out of scope").
4. **Version-history wiring.** Same controller listens for
   `wave-nav-version-history-requested` on each nav-row host and
   toggles the document's `<wavy-version-history>` overlay's `open`
   property (`element.open = true`; the element reflects to the
   `open` attribute and removes `hidden` in `_syncOpen`). The
   overlay's own Esc handler already closes it. This satisfies the
   issue's "click history → overlay opens; Esc → overlay closes"
   acceptance. Single-writer: nothing else writes `open` on
   `<wavy-version-history>` today.
5. **Search-rail re-query.** When the controller emits
   `wavy-folder-action-completed`, the search rail card host's
   bootstrap script re-fires the current saved-search query so
   `in:pinned` / inbox digest cards reflect the new state without a
   page reload. Implementation: dispatch the event on `document` and
   add a one-line bootstrap listener that flips a
   `data-folder-action-pending` attribute on `[data-j2cl-search-rail]`
   then re-runs the rail's existing `refresh()` if the host exposes
   it (most do via `wavy-search-rail`'s `requestUpdate()`-aware
   query loader). When the rail does not expose `refresh()`, we
   fall back to dispatching a `wavy-search-refresh-requested` event
   it already listens for in subsequent slices. The E2E does not
   depend on the rail showing the new pinned card — it independently
   navigates to `?q=in:pinned` to verify, as GWT does.

## Acceptance

1. `wavy-wave-nav-row` renders SVG icons matching GWT
   `ViewToolbar.java` glyphs; visible tooltips via `title`.
2. New Playwright E2E `wave-actions-parity.spec.ts`:
   - Register fresh user, the WelcomeRobot seeds an inbox wave.
   - On `?view=j2cl-root`:
     - Open the welcome wave by clicking its rail card.
     - Click pin (await one `wavy-folder-action-completed` event so
       the test does not race the search-index sync); assert
       `wavy-wave-nav-row[pinned]`; navigate to
       `?view=j2cl-root&q=in:pinned` → assert the wave appears.
     - Click pin again → row no longer `pinned`; pinned query no
       longer shows the wave.
     - Click archive → await completion event; wave's title absent
       from inbox digest list.
     - Click archive again → title back in inbox.
     - Click history → `<wavy-version-history>` no longer has
       `hidden` and shows its panel. Press Esc → `hidden` re-applied.
   - On `?view=gwt`: same archive + pin assertions via the GWT
     toolbar buttons. The GWT toolbar renders SVG icons inside
     `[aria-label*='Pin'|'Unpin'|'archive'|'history']` controls. If
     the GWT half lacks robust selectors for one of these, the
     test asserts the GWT click route triggered the same
     `POST /folder?operation=...` request via Playwright's network
     interceptor (GWT integration tests already cover the server
     side; this Playwright run only proves the click reaches the
     wire).
3. Manual verification log appended to PR body listing every action
   exercised.

## Risks / tradeoffs

- **Selector drift in nav-row tests.** The existing
  `wavy-wave-nav-row.test.js` reads button text content (e.g.
  "Archive" / "Pin"). Replacing labels with icons breaks those text
  assertions. Mitigation: keep `aria-label` and `data-action` on
  every button (they already exist) and update tests to assert on
  `aria-label` / `data-action` instead of text. This is a
  one-character delta per assertion and matches how the format
  toolbar tests already query.
- **Folder servlet feature-detect.** `FolderOperationServlet` ships
  in the Java tree (`/folder` route). Local smoke confirms it
  responds for both views. Test will assert HTTP 200 once and rely
  on the search rail's own re-query for the visible side-effect.
- **GWT half flakiness.** GWT renders the action toolbar lazily. The
  G-PORT-4 spec already pre-establishes the `[aria-label*='participant']`
  pattern; this slice extends it to the archive/pin labels with
  poll-based waits.
- **No legacy fallback.** Per project memory
  `feedback_no_legacy_fallbacks_for_flagged_features`, the new
  controller is unconditionally active on `?view=j2cl-root`. No
  feature flag.

## Test plan

- Local: register a fresh user, exercise each button on both views,
  check rail card behavior + version-history overlay open/close.
- Unit: `npm test --prefix j2cl/lit` (Web Test Runner — covers
  `wavy-wave-nav-row` + the new controller).
- E2E: `npx playwright test wave-actions-parity.spec.ts` from
  `wave/src/e2e/j2cl-gwt-parity/`.
- Server: existing `HtmlRendererJ2clRootShellIntegrationTest`
  remains green (asserts presence of nav-row + version-history
  mount).

## Out of scope

- Restoring a wave from version history (deferred — F-2 follow-up
  #1054). The Restore button's `restoreEnabled` gate stays at its
  current `false` default in `<wavy-version-history>`.
- Toast notifications for folder-op failure. The controller emits a
  `wavy-folder-action-failed` event and stops there.
- Refresh button — see "Refresh button: explicitly out of scope"
  above.
- Changing `<wavy-search-rail>` data flow beyond the new `refresh()`
  call.
