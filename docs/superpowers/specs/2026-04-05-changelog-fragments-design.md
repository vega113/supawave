# Changelog Fragment System Design

**Date:** 2026-04-05
**Status:** Approved
**Branch:** feat/changelog-fragments

## Problem

All PRs modify `wave/config/changelog.json` (59 entries, 996 lines). The file must be in newest-first order. When many PRs are open simultaneously, they all insert at position 0, causing constant merge conflicts and CI failures:

```
new releaseIds must be prepended above existing history; found '...' after base entries began
```

The `validate_against_base()` function in `validate-changelog.py` enforces that new entries must be prepended above all existing base entries. This makes concurrent PRs fundamentally incompatible.

## Solution

Replace the monolithic `changelog.json` with per-release fragment files.

### Source of Truth

Each changelog entry lives in its own file:

```
wave/config/changelog.d/{releaseId}.json
```

The JSON schema is identical to a single entry in the old changelog.json array:

```json
{
  "releaseId": "2026-04-05-example-feature",
  "version": "PR #123",
  "date": "2026-04-05",
  "title": "Example feature title",
  "summary": "One-line summary of the change.",
  "sections": [
    {
      "type": "feature",
      "items": ["Added X", "Added Y"]
    }
  ]
}
```

### Generated Artifact

`wave/config/changelog.json` is always generated by `scripts/assemble-changelog.py` and never manually edited. It is added to `.gitignore`.

### Assembler: `scripts/assemble-changelog.py`

- Reads all `*.json` files from `wave/config/changelog.d/`
- Validates each fragment matches the entry schema
- Sorts entries by `(date DESC, releaseId DESC)`
- Writes the assembled JSON array to `wave/config/changelog.json` (or stdout with `--stdout`)
- Idempotent: running it twice produces the same output

### Validator: `scripts/validate-changelog.py`

Rewritten to validate fragment files only. All monolithic-format logic removed:

1. Each fragment in `changelog.d/` is valid JSON matching the entry schema
2. Filename (minus `.json`) matches the `releaseId` field inside
3. No duplicate releaseIds across fragments
4. The assembled output has valid date ordering (newest-first)
5. **Removed**: `validate_against_base()`, `--base-ref` flag, `load_base_changelog()`

### CI Changes

#### `build.yml` — PR and push validation

```
1. Run assemble-changelog.py → generates wave/config/changelog.json
2. Run validate-changelog.py --changelog wave/config/changelog.json
3. sbt build (copies generated changelog.json to target)
4. diff -u wave/config/changelog.json target/universal/stage/config/changelog.json
```

The `--base-ref` argument is removed from all CI invocations.

#### `deploy-contabo.yml` — Deploy workflow

```
1. Run assemble-changelog.py → generates wave/config/changelog.json
2. Run validate-changelog.py (no --base-ref)
3. sbt build
4. diff check
5. Email extraction reads generated changelog.json
```

### build.sbt

No change. The `resourceGenerators` task copies `wave/config/changelog.json` which the assemble step generates before `sbt` runs.

### Migration

One-time, done in this PR:

1. Split all 59 existing entries from `changelog.json` into individual fragment files in `changelog.d/`
2. Delete `wave/config/changelog.json` from git tracking
3. Add `wave/config/changelog.json` to `.gitignore`
4. Verify round-trip: assemble fragments → output matches original changelog.json content

### After This PR

- **Only one way** to add a changelog entry: create a file in `wave/config/changelog.d/`
- No merge conflicts on changelog entries (each PR owns a uniquely named file)
- `validate_against_base` ordering constraint is gone
- The assembled `changelog.json` is a build artifact, not a source file
